# 起报点 与 滚动回测（入门概念说明）

> 这份文档用大白话解释两个评估时序预测必须懂的核心概念：**起报点**和**滚动回测**。
> 它们是实验手册 §4「评估协议」的基础。名字听着唬人，道理其实很朴素。

---

## 一、起报点（forecast origin）

### 一句话定义

**起报点 = 你假装站在某个时刻，开始往后预测的那个时间点。**

### 打个比方

想象你是 2025 年 8 月 1 日早上 0 点的电力调度员。你手里只有**这一刻之前**的历史数据，你要预测接下来 24 小时（8 月 1 日全天）的电价。这个"你站着的时刻"——8 月 1 日 0 点——就叫**起报点**。

### 它把时间轴切成两半

```
        ←──── 历史（能看）────┤起报点├──── 未来（要预测）────→
                            8月1日0点
        [    上下文窗口    ]         [    预测窗口    ]
        比如往前 168 小时             比如往后 24 小时
```

- **起报点左边**：模型能看到的"上下文"（历史数据）。
- **起报点右边**：模型要预测、且**预测时绝对不能偷看**的"答案"。

等真实数据到了，再拿预测和答案对比，算误差。

### 两个相关参数

- **上下文长度 (context length)**：起报点往左取多长的历史给模型看。比如 168 小时 = 过去 7 天。
- **预测步长 (horizon)**：起报点往右预测多远。比如 24 小时 = 未来 1 天。

---

## 二、当前做法的问题：只有一个起报点

项目现有代码（`iloc[-PRED_LEN:]`）只做了**一个**起报点——数据的最后一天。相当于只考了模型一道题：

> "请预测最后一天的电价。" 模型答完，算个 MAE，完事。

**这有个致命问题：一道题的成绩说明不了谁更强。**

- 万一这天天气平稳、电价规律 → 所有模型看起来都不错；
- 万一这天刚好有尖峰 → 可能所有模型都翻车；
- 换一天考，模型排名可能完全颠倒。

结论全凭运气，写进报告/论文是**站不住脚的**。

---

## 三、解决办法：滚动回测（rolling backtest / walk-forward）

> 注：口语里有时会说成"回滚"，**正确术语是「回测」或「滚动回测」**。

### 一句话定义

**滚动回测 = 不只考一道题，而是沿时间轴设很多个起报点，让模型一道接一道连着考，最后看平均分。**

### "滚动"是什么意思

起报点像滚轮一样，沿时间往前一格一格挪：

```
起报点1：站在 8月1日0点  → 预测 8月1日全天  → 算误差 ①
起报点2：站在 8月2日0点  → 预测 8月2日全天  → 算误差 ②
起报点3：站在 8月3日0点  → 预测 8月3日全天  → 算误差 ③
   ...                    ...                 ...
起报点30：站在 8月30日0点 → 预测 8月30日全天 → 算误差 ㉚

最终成绩 = 这 30 个误差的平均值 ± 标准差
```

每挪一格（每过 STRIDE 小时，通常 24h），模型能看到的历史就多一段，要预测的也是新的一段。

### 一条不能违反的铁律

**每个起报点，模型都只能看自己左边的数据，右边的答案永远盖住。**

这就是"回测"二字的由来——拿历史数据**模拟**"如果当时真的这样预测，会不会准"。需要训练的模型（RF/XGBoost/LSTM）的 `fit`，也只能用起报点左边的数据，绝不能用到右边或未来，否则就是**数据泄露**，结论作废。

---

## 四、为什么滚动回测就靠谱了

考 30 天 = 横跨工作日和周末、平稳日和尖峰日、各种天气。这时候能得出真正有意义的结论：

| 看什么 | 说明什么 |
|---|---|
| 谁的**平均分**高 | 谁整体更强 |
| 谁的分数**波动小**（标准差小） | 谁更稳定、更可靠 |
| 单独看"尖峰日"那几次 | 谁的 Spike-F1 高 → 谁尖峰预警能力强 |

于是"A 比 B 好"就从一句运气话，变成了**有统计意义的可信结论**。

## 五、伪代码：滚动回测长什么样

本质就是一个让起报点在时间轴上一格格往前挪的循环：

```python
results = []
t = test_start
while t <= test_end:
    # 起报点 t：只取它左边的历史
    context = data[t - CONTEXT_LEN : t]      # 模型能看的上下文
    actual  = data[t : t + HORIZON]          # 真值（仅用于事后算误差）

    for model in models:
        pred = model.predict(context, horizon=HORIZON)   # 预测时看不到 actual
        err  = evaluate(pred, actual)                    # MAE / RMSE / Spike-F1 ...
        results.append((t, model.name, err))

    t = t + STRIDE        # 起报点向前挪一格（通常 24 小时）

# 汇总：每个模型在所有起报点上的平均成绩 ± 标准差
summary = aggregate(results)
```

## 六、和实验手册的关系

- 本文是实验手册（`docs/specs/experiment_manual.md`）**§4 评估协议**的入门补充。

- 推荐默认参数：测试期覆盖 4–8 周、STRIDE = 24h、起报点 ≥ 30 个、上下文 168h、步长 24h（这些都是可在消融中调整的"旋钮"）。

## 七、一句话总结

- **起报点**：假装站在某个时刻往后预测的那个时间点。
- **滚动回测**：沿时间设很多个起报点连续考，看平均成绩——让结论从"靠运气"变成"靠统计"。
